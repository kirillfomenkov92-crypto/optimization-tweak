# План: углубление архитектуры (топ-находки аудита)

> Создано скиллом **writing-plans** на основе отчёта **improve-codebase-architecture**.
> Принципы: DRY, YAGNI, **TDD** (тест до кода), частые коммиты, маленькие задачи.
>
> **Важно:** ни одна задача не должна менять наблюдаемое поведение инструментов
> для пользователя. Каждая правка — рефакторинг под прикрытием тестов. GUI вживую
> не тестировался, поэтому UI-затрагивающие задачи (#7, #8) отложены до живого
> прогона на Windows.

## Контекст для исполнителя

Репозиторий содержит два независимых инструмента на Python:
- `windows_optimizer/` — PyQt6-приложение; модули в `app/modules/*`, ядро в `app/core/*`.
- `turbo_debloat/` — CLI/GUI деблоатер; движок в `core/engine.py`, защита в `core/safety.py`.

Оба собираются в `.exe` только в GitHub Actions (Windows). Локально (Linux) PyQt6
не установлен — проверяются компиляция, не-Qt импорты и тесты на `assert`
(см. `turbo_debloat/tests/`, `windows_optimizer/tests/`). Запуск тестов:
`python <путь>/test_*.py` из корня репозитория, ожидается код выхода 0.

## Карта файлов (что создаём/трогаем)

| Файл | Ответственность | Действие |
|------|-----------------|----------|
| `windows_optimizer/app/core/tweak_applier.py` | единый apply/verify твика реестра | создать |
| `windows_optimizer/tests/test_tweak_applier.py` | тесты помощника | создать |
| `app/modules/{network,gaming,cpu,power,gpu,memory}.py` | заменить тело apply на вызов помощника | править |
| `windows_optimizer/app/core/result.py` | тип `OperationResult` | создать |
| `windows_optimizer/tests/test_operation_result.py` | тесты типа | создать |

`shared/`-логгер (находка #5) намеренно **не** трогаем: два инструмента собираются
в отдельные `.exe`, общий пакет усложнит сборку PyInstaller. Дублирование 55 строк
дешевле, чем общая зависимость между двумя независимыми артефактами — **YAGNI**.

## Задача 1 — `RegistryTweakApplier` (находка #1)

**Цель:** один помощник, который читает старое значение, пишет новое, читает обратно
для верификации и логирует — вместо 6 копий в модулях.

### Шаги (TDD)
1. **Тест сначала.** Создать `windows_optimizer/tests/test_tweak_applier.py`:
   - замокать `app.utils.registry_helper` (read_value/write_value) на словарь в памяти;
   - проверить: `apply_one` пишет значение, возвращает `True`, когда read-back совпал;
   - проверить: при несовпадении read-back возвращает `False` и пишет `WARN`;
   - проверить: при исключении из write_value возвращает `False` (не пробрасывает);
   - `apply_many` возвращает `{name: bool}` для списка.
   Запустить — тест **падает** (модуля ещё нет). Это обязательный шаг TDD.
2. **Код.** Создать `app/core/tweak_applier.py` с `apply_one(hive,path,name,rtype,value)->bool`
   и `apply_many(tweaks)->Dict[str,bool]`. Сигнатура кортежа твика —
   `(hive, path, name, rtype, value)`. Логировать через существующий `log_change`.
3. Прогнать тест — **зелёный**.
4. **Миграция по одному модулю за коммит.** Начать с `network.py`:
   - привести `TCP_TWEAKS` к форме `(hive,path,name,rtype,value)` (добавить `"HKLM"`);
   - заменить тело `apply_tcp_tweaks` на `return RegistryTweakApplier.apply_many(...)`;
   - НЕ удалять колонки `default/desc`, если их использует `scan()` — оставить
     отдельную структуру для scan или расширить кортеж осознанно.
   - прогнать существующие тесты + компиляцию, **коммит**.
5. Повторить п.4 для `gaming.py`, `cpu.py`, `power.py`, `gpu.py`, `memory.py` —
   **по одному коммиту на модуль**, чтобы ревью было дёшево, а откат точечным.

### Проверка
`python windows_optimizer/tests/test_tweak_applier.py` → 0; компиляция всего `app`.
Поведение не меняется: те же значения пишутся в те же ключи.

## Задача 2 — `OperationResult` (находка #3)

**Цель:** различать «пропущено ради защиты» и «ошибка». Сейчас `set_start_type`
возвращает `False` в обоих случаях.

### Шаги (TDD)
1. **Тест сначала.** `windows_optimizer/tests/test_operation_result.py`:
   - `OperationResult(success=True, skipped=True)` → `is_ok()` True, но `skipped`;
   - `OperationResult(success=False, error=e)` → `is_ok()` False;
   - сериализация в dict для UI стабильна.
2. **Код.** `app/core/result.py` с `@dataclass OperationResult` (поля:
   `success, message="", skipped=False, detail=None, error=None, undo=[]`) и `is_ok()`.
3. **Точечное внедрение, не массовое.** Первым перевести только `services.set_start_type`:
   - защита → `OperationResult(success=True, skipped=True, message="защищено")`;
   - успех/ошибка → соответствующий результат.
   - обновить вызыватель в UI (`services_panel`), чтобы показывать «пропущено» отдельно.
   - **коммит**. Остальные модули — отдельными задачами позже (YAGNI: не переписываем
     всё сразу без потребности).

### Проверка
Тесты зелёные; `services_panel` корректно показывает пропуск vs ошибку.
Откладывается до подтверждения, т.к. затрагивает UI (живой прогон желателен).

## Что НЕ делаем сейчас (и почему)

- **#5 общий logger** — YAGNI/риск сборки двух `.exe`. Оставляем дубль.
- **#7 ScanRow, #8 оркестратор, #4 трейты** — затрагивают UI и поток выполнения;
  без живого GUI-теста на Windows риск регрессии выше пользы. Вернуться после того,
  как пользователь прогонит `.exe` на Windows и пришлёт скриншот.
- **#2 единая safety-политика** — базовый риск уже снят security-фиксом
  (регистронезависимость + тесты). Полное слияние двух независимых инструментов —
  отдельное продуктовое решение, не рефакторинг.

## Порядок и коммиты
1 коммит = тест+помощник (Задача 1, п.1–3) → затем по коммиту на модуль (п.4–5) →
Задача 2 отдельной веткой/PR после согласования UI-части. Каждый PR гоняет CI
(сборка `.exe` + тесты) и мержится только зелёным.

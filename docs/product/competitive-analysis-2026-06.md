# Конкурентный анализ и план фич — Windows Optimizer Pro

> Продукт-документ. Цель: взять лучшие фичи у конкурентов и отобрать ТОП-5,
> реализуемых в нашем стеке (Python 3.11 + PyQt6, один `.exe`, **без бэкенда и
> БД**, локально), под нашу аудиторию (нетехнические пользователи слабых ПК) и
> наш принцип «безопасность прежде всего».
>
> Источник списка (`amssoft.ru/.../optimizaciya-raboty-kompyutera.php`) отдал 403,
> поэтому анализ построен на стандартном наборе конкурентов этой категории.
> Если нужен другой набор — заменить и пересчитать.

## 2. Отбор: что реально подходит нам

Критерии: (а) польза нетех-пользователю, (б) реализуемо локально без сервера/БД,
(в) не противоречит «безопасность прежде всего», (г) даёт конкурентное отличие.

**Осознанно НЕ берём** (важно для позиционирования):
- ❌ **Драйвер-апдейтер / софт-апдейтер** (IObit, Auslogics) — требуют интернета,
  каталога версий и несут реальный риск поломки. Против нашего принципа и стека.
- ❌ **Чистка реестра «для скорости»** (CCleaner и др.) — эффект на скорость
  мифический, риск > пользы. Наше отличие — **не делать вредных плацебо-фич**.
- ❌ **Облако/аккаунты/телеметрия** — мы принципиально офлайн.

Это не упущение, а позиция: **доверие и безопасность вместо фич-булимии.**

### Фича 1 — One-Click «Ускорить» (Smart Optimize) · High · 4–6 д

**Зачем:** наша ЦА не пойдёт по 15 разделам. Нужен один большой безопасный
сценарий (как у Advanced SystemCare / Ускорителя): нажал → увидел план → применил.

**UX в нашем приложении:**
- На Дашборде крупная кнопка **«⚡ Ускорить компьютер»**.
- Клик → экран **предпросмотра (dry-run)**: «Будет сделано N безопасных
  улучшений» сгруппировано по понятным категориям, с риск-бейджами 🟢.
- Кнопка **«Применить»** → создаётся точка восстановления/бэкап → прогресс →
  отчёт «Готово: применено X, пропущено Y» и обновлённый Health Score.
- В набор входят ТОЛЬКО 🟢-безопасные действия из уже существующих модулей
  (твики из БД с `risk_level=safe`, безопасные службы, очистка временных файлов,
  безопасные сетевые/игровые твики). Деблоат **не** включаем в one-click
  (он необратим по приложениям) — он остаётся отдельным осознанным разделом.

**Технический план:**
1. `app/core/smart_optimize.py`: собрать список `Operation` из существующих
   модулей, фильтруя по `risk_level == "safe"` / белому списку. Переиспользовать
   `RegistryTweakApplier`, `OperationResult`, `ServicesModule` (только `safe`),
   `DiskModule` (безопасные категории).
2. Выполнение — через существующий `StepWorker` (поток, прогресс, `stop_on_error`)
   и `backup.create_backup` перед применением (как уже сделано в панелях).
3. UI: новый экран в Дашборде (или `app/ui/smart_optimize_panel.py`) на базе уже
   существующего паттерна dry-run → apply → отчёт (как в панели деблоата).
4. БД/состояние: не требуется. Список «что входит» — конфиг в коде/JSON.

**Риски:** один из твиков может не примениться на конкретной машине →
митигируется верификацией `RegistryTweakApplier` и отчётом по каждому пункту.
Бэкап обязателен. **Без дисплея не протестировать визуально** — нужен прогон на
Windows.

### Фича 2 — Health Score с «до/после» · High · 2–3 д

**Зачем:** у IObit/Wise виден «рейтинг здоровья» — он мотивирует и показывает
результат. У нас `health_score()` уже есть, но не сведён и не показывает динамику.

**UX:** на Дашборде кольцо/число 0–100 с подписью («Хорошо»/«Можно лучше»).
После Smart Optimize — анимация роста и **дельта** («было 62 → стало 81»).

**Технический план:**
1. `app/core/health.py`: агрегатор — собрать частные `health_score()` модулей
   (реестр, службы, автозагрузка, диск) во взвешенную оценку 0–100 + список
   «что снижает оценку».
2. Дашборд: виджет-кольцо (`QPainter`, как уже рисуется иконка трея) + хранение
   предыдущего значения в `QSettings` для дельты.
3. Пересчёт после применения (сигнал от `StepWorker.finished_ok`).

**Риски:** оценка не должна «врать» — считаем по доле применённых безопасных
улучшений, без маркетинговых «99 проблем». Честность = наше отличие.

### Фича 3 — Оценка влияния автозагрузки · Medium · 3–4 д

**Зачем:** CCleaner/Task Manager показывают «влияние на запуск». Новичку это
понятнее, чем просто список программ автозапуска (который у нас уже есть).

**UX:** в разделе «Автозагрузка» у каждого пункта бейдж **Высокое / Среднее /
Низкое** влияние на загрузку + кнопка «Отключить» с откатом.

**Технический план:**
1. Источник данных — `StartupApproved` и значения из
   `HKCU/HKLM\...\Explorer\StartupApproved\*` (Windows хранит там
   замеры/состояние) и/или измерение через `Get-CimInstance Win32_StartupCommand`
   + эвристика по известным «тяжёлым» процессам.
2. `app/modules/startup.py`: расширить `scan()` полем `impact`.
3. UI: дополнить плейсхолдер «Автозагрузка» бейджами (наш `format_scan_row`/
   риск-бейджи переиспользуемы).

**Риски:** точный «boot delay» Windows не всегда отдаёт сторонним приложениям →
используем приближённую градацию, честно подписанную как оценка.

### Фича 4 — Планировщик обслуживания · Medium · 3–4 д

**Зачем:** Auslogics/Glary умеют «обслуживать сами раз в неделю». Для нашей ЦА
(«поставил и забыл») это ценно: ПК поддерживается без участия.

**UX:** в Настройках — переключатель **«Автообслуживание: раз в неделю»** +
выбор «только чистка» / «чистка + безопасные твики». Тихо, с уведомлением-итогом.

**Технический план:**
1. Реестрируем задачу через **Windows Task Scheduler** (`schtasks /create`) — мы
   уже работаем с `schtasks` в движке деблоата, инфраструктура есть.
2. Задача запускает `.exe` со скрытым флагом `--maintenance` → выполняет
   Smart Optimize (Фича 1) в тихом режиме, пишет в историю/лог.
3. Состояние (вкл/выкл, периодичность) — в `QSettings`.

**Риски:** тихие изменения системы должны быть только 🟢-безопасными и с бэкапом;
пользователь должен явно включить (по умолчанию выкл). Без сервера — всё локально.

---

### Фича 5 — Чистый деинсталлятор (с зачисткой остатков) · Medium · 5–7 д

**Зачем:** IObit Uninstaller популярен тем, что удаляет программу **и** подчищает
остаточные файлы/ключи. Реальная польза на слабом/захламлённом ПК.

**UX:** новый раздел «Удаление программ»: список установленного (имя, размер,
дата) → «Удалить» → запускается штатный анинсталлер → затем сканер остатков
(папки в `Program Files`/`AppData`, осиротевшие ключи реестра) → показывает
найденное → удаляет **после подтверждения и бэкапа реестра**.

**Технический план:**
1. Список программ: `HKLM/HKCU\...\Uninstall\*` (`DisplayName`,
   `UninstallString`, `InstallLocation`, `EstimatedSize`).
2. Удаление: запуск `UninstallString` через `subprocess` (список аргументов).
3. Скан остатков: сопоставление по имени/издателю в известных местах + проверка
   «ключ ссылается на несуществующий путь». Перед удалением — `backup.create_backup`.
4. UI: панель по образцу существующих (scan → подтверждение → применить → откат).

**Риски:** самый рискованный пункт (можно задеть нужное) → строгая эвристика,
**обязательное подтверждение и бэкап**, белый список «не трогать». Поэтому
Medium-приоритет и максимум тестов на реальной системе.

## 4. Дорожная карта (3 месяца)

```
Месяц 1  — «Ускорить в один клик» (главный рывок UX)
  Нед.1–2  Фича 1: Smart Optimize (ядро + панель dry-run/apply/отчёт)
  Нед.3    Фича 2: Health Score «до/после» на Дашборде
  Нед.4    Тестирование на реальной Windows + полировка, релиз 2.1

Месяц 2  — «Прозрачность и автономность»
  Нед.5–6  Фича 3: Оценка влияния автозагрузки
  Нед.7    Фича 4: Планировщик обслуживания (тихий --maintenance)
  Нед.8    Тестирование + релиз 2.2

Месяц 3  — «Глубокая чистка»
  Нед.9–11 Фича 5: Чистый деинсталлятор (с бэкапом и подтверждением)
  Нед.12   Сквозное тестирование, обновление ДЛЯ_ДРУГА/README, релиз 2.3
```

**Принцип этапности:** сначала то, что даёт максимум ценности при минимуме риска
и переиспользует имеющийся код (1, 2), затем — фичи прозрачности/автономности
(3, 4), и в конце — самую рискованную (5) с упором на тестирование.

## 5. Главная рекомендация

Не гнаться за числом фич. Наше конкурентное преимущество против CCleaner/IObit —
**безопасность, честность и простота**: один понятный сценарий «Ускорить», видимый
честный результат, никаких плацебо (чистка реестра «для скорости») и опасных
апдейтеров драйверов. Это и есть позиционирование «оптимизатор, которому не
страшно отдать ПК неопытному другу».
